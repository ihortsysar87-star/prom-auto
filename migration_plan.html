1. What the n8n workflow actually does

One Telegram bot, two independent jobs sharing it. The first is the live product-listing pipeline, triggered by a photo message. The second is a bulk Prom.ua → MySQL sync that has no trigger node in the export — it was run by hand from the n8n editor, not on any schedule.

Three nodes — Parse raw data, Send a text message, Edit Fields — sit in the canvas fully disconnected from the trigger. They read like an earlier draft of the reply-formatting step, superseded by parse to prom api. Not being carried into the Python port unless you tell me otherwise.

4. Secrets — do not carry these over as-is

Carried over from the n8n export

The workflow JSON had three live credentials hardcoded directly in node parameters: a Prom.ua bearer token, an OpenAI key, and the Telegram bot token embedded in plain HTTP URLs. All three are now read from environment variables via prom_auto/config.py and .env (git-ignored) — but the old values are still valid until you rotate them.

Before switching real traffic to the Python bot: rotate the Prom.ua API token, the OpenAI key, and the Telegram bot token, then put only the new values in .env.

5. Open questions

  • Model — resolved. gpt-5.4 from the n8n export turned out to be a real, current OpenAI model, not a typo. After comparing cost and capability across gpt-4o-mini, gpt-5.4, gpt-5.4-mini, and gpt-5.4-nano, config.py now defaults to gpt-5.4-mini — full vision + web-search tool support at a fraction of the full model's cost, on a generation OpenAI isn't sunsetting (unlike gpt-4o-mini).
  • Image hosting — tried web search, settled on Telegram photos. The n8n flow pointed Prom.ua's Посилання_зображення field at a Telegram api.telegram.org/file/bot<token>/… URL, which expires and leaks the bot token to anyone who views the listing's image source. We tried replacing it with a real product photo from the web_search tool's image results (search_content_types: ["image", "text"], even forcing the tool with tool_choice: "required"), but across every live test gpt-5.4-mini only ever returned text_result entries — never image_result — so the feature was dropped. telegram_bot.py now just uses the user's own Telegram photo(s) directly (still the same expiring-URL tradeoff as the original n8n flow, just no longer masked by an image-search feature that wasn't actually working).
  • Multiple photos per listing — added. Telegram sends an album as several separate updates sharing one media_group_id. telegram_bot.py now buffers those for ~1.5s after the last one arrives, then processes the whole album as a single product: all photos go to openai_client.identify_product() for identification, and all of them are joined with ; into Prom.ua's Посилання_зображення field (its documented separator for multiple images). A single photo with no album still works exactly as before.
  • Prom.ua import headers — fixed. The first live test failed with Невідомі заголовки: 'Бренд', 'Матеріал', 'Модель' — those three aren't valid columns for this Prom.ua account's import template (Prom.ua ignores/rejects any header not in its own documented list, custom characteristics aren't just free-form column names). product_mapper.py now folds brand/model/material into the description text instead of sending them as standalone columns.
  • Region — added. Prom.ua's XLS import field for the item's location is Де_знаходиться_товар (not a "region" column), taking a locality name. Added as config.PROM_REGION (default Київ) and wired into every listing.
  • Job B, if it comes back into scope. It had no trigger node at all in the export, and the target MySQL products table's schema isn't in the workflow export either — both would need answers before porting it.
  • Retry behavior. The n8n upload node retried up to 5 times with a 5s wait; prom_client.import_file() doesn't retry yet. Add it if Prom.ua's endpoint is flaky in practice.

6. Rollout checklist

  1. done
    Scaffold the package. Folder structure, config loader, and a first-pass implementation of every module in the mapping table above.
  2. done
    Resolve the open questions. Model chosen (gpt-5.4-mini), secrets rotated, image-hosting approach settled (Telegram photos), Prom.ua headers and region field fixed against real import errors.
  3. done
    Dry-run Job A. Multiple real photos (single and multi-photo albums) sent through telegram_bot.py against the live Prom.ua catalog — listings created successfully each time.
  4. later
    Cut over. Deactivate the n8n workflow (already inactive) and run the Python bot as the long-lived process (systemd / Task Scheduler / similar) instead of an ad-hoc foreground session.
